# Analyses of things to fix

Here's a detailed breakdown of the shortcomings I found, organised by category:

---

## 1. Spec vs. Implementation Mismatches

### Navigation Strategy
The spec mandates **bottom tabs** as the primary navigation method with a **center action button** spawning a **bottom sheet** for secondary navigation. Neither portal implements this:
- **Customer** uses a `Drawer` (side menu), which is not mentioned in the spec at all.
- **Technician** uses a plain `Stack`, with no tabs or action button whatsoever.

### Missing `app/manager/_layout.jsx`
The spec explicitly calls for a manager layout with a blank "login working but not implemented" screen and a logout button. The `app/manager/` directory does not exist at all.

### Missing `app/debug/_layout.tsx`
The spec calls for a debug master switch layout at `app/debug/`. It does not exist.

### Missing Middleware and Guard Hooks
The spec calls for:
- `middleware.ts` — route-level protection
- `useRoleGuard.ts` — component-level enforcement
- `useGuestGuard.ts` — redirect logged-in users away from auth routes

None of these exist. The guard logic is duplicated inline in each `_layout.jsx` instead.

### `resolveHomeRoute` Doesn't Handle All Roles
In `UserContext.jsx`, `resolveHomeRoute` only handles `technician` and falls back to `/customer/home` for everything else. The spec defines `owner` and `manager` roles too. An owner logging in would incorrectly be sent to the customer portal.

```caact-mobile/context/UserContext.jsx#L218-224
const resolveHomeRoute = (user) => {
    if (!user) return \"/sign-in\";
    const normalized = normalizeUser(user);
    if (String(normalized.status || \"active\").toLowerCase() !== \"active\") {
      return \"/sign-in\";
    }
    if (normalized.role === \"technician\") return \"/technician\";
    return \"/customer/home\";
  };
```

### `recover/factor` is Not a Bottom Sheet
The spec says `/recover/factor` should be a **bottom sheet**. It is implemented as a full-screen `Stack` page.

### `task/[id]/unit/log/insert` is Not a Bottom Sheet
The spec says this route should be a **bottom sheet** (with draft-save-on-close behaviour). It is a full-screen `ScrollView` page. The draft-save-on-close is partially implemented (via `persistDraftAndBack`), but only when the user explicitly presses "Save Draft and Close" or the back button — not on any other close gesture.

### `task/[id]/unit/log/delete` is Not a Modal Prompt
The spec says this should be a **modal prompt**. It is implemented as a dedicated screen that immediately fires an `Alert` on mount — a reasonable workaround, but architecturally it is a screen, not a modal route.

### `customer/oobe` Missing TOTP QR Display
The spec says the customer OOBE should display the **TOTP secret in text and QR form**. The `oobe/index.jsx` shows the TOTP secret as plain text only — there is no QR code rendered.

### `technician/oobe/reset.jsx` Uses a Hardcoded Fake Secret
```caact-mobile/app/technician/oobe/reset.jsx#L15-15
  const debugSecret = \"TECH-DEBUG-TOTP-SECRET\";
```
The customer version correctly fetches the real TOTP secret from `customerSecurityService`. The technician reset screen uses a hardcoded placeholder string and never reads from storage, making it non-functional.

### `recover/factor/2.jsx` — Recovery Code Logic is Stub/Broken
The spec says recovery code entry should allow the user to sign in. The implementation accepts any 6-digit number, never validates it against stored recovery codes, and just shows a success alert that tells the user to "login with your new password" — which makes no sense for a recovery code flow. The `login` import is unused.

### `recover/factor/1.jsx` — Password Reset is Incomplete
The screen generates a reset code and shows it in a debug card, but there is no subsequent step to actually enter the code and set a new password. The flow dead-ends.

### Sign-Up Step 1 TOTP Verification is Not Real
```caact-mobile/app/(auth)/sign-up/step/1.jsx#L114-122
  const handleVerifyTotp = () => {
    if (!/^\d{6}$/.test(totpInput.trim())) {
      Alert.alert(\"Invalid Code\", \"Enter a valid 6-digit TOTP code.\");
      return;
    }
    router.push({ ... });
  };
```
The TOTP code entered by the user is never validated against the generated `totpSecret`. Any 6-digit number passes verification.

### `app/(auth)/sign-up.jsx` Route Conflict
There is both `app/(auth)/sign-up.jsx` (a redirect shim) and `app/(auth)/sign-up/step/0.jsx`. The `_layout.jsx` registers both `sign-up` and `sign-up/step/0` as separate screens, which is redundant and potentially confusing for Expo Router's file-based routing.

### `app/(auth)/login.jsx` vs `sign-in.jsx` Duplication
`sign-in.jsx` is just `export { default } from "./login"`. The spec only defines `/sign-in`. The `/login` route is an undocumented alias that also appears in navigation calls (e.g. `router.push("/login")` in recover screens), creating inconsistency.

## 2. `UserContext` / Architecture Issues

### Context Still Exposes `users` Array But Doesn't Populate It
`users` is exposed in the context value and used in `recover/factor/0.jsx` and `factor/2.jsx` to look up users by email. But `fetchUsers` is never called automatically — `users` will always be `[]` unless a caller explicitly invokes `fetchUsers`. The recovery screens will always fail to find any user.

### `storageKeys.jsx` is Stale / Unused
`services/storageKeys.jsx` still contains `GOOGLE_USER` and uses different key names (`"users"`, `"currentUser"`) than `authStorage.jsx` (`"auth_users"`, `"auth_current_user"`). The spec says Google auth is removed. This file appears to be a leftover that is no longer used.

### `authStorage.jsx` is Unused
Now that `UserContext` delegates everything to `api.jsx` (the HTTP backend), `authStorage.jsx` is dead code. It still defines its own `STORAGE_KEYS` with the correct spec-aligned key names, but nothing calls it.

### `api.jsx` Points to `localhost:5050` — Backend Doesn't Exist
The spec explicitly states: *"backend: none (fully local, no caact-server code present)"* and *"architecture: fully local/offline"*. Yet `UserContext` calls `api.jsx` which makes HTTP requests to `http://localhost:5050`. This means **login, registration, and session hydration are all broken** unless a server happens to be running. The entire auth layer contradicts the spec.

### `clearAuditLogs` is a No-op
```caact-mobile/context/UserContext.jsx#L208-210
  const clearAuditLogs = async () => {
    return [];
  };
```
This function does nothing and returns an empty array. It's exposed in the context but non-functional.

## 3. Technician Portal Issues

### `technician/home.jsx` Uses `useFocusEffect` Without `useCallback` Import
```caact-mobile/app/technician/home.jsx#L20-21
  useFocusEffect(
    React.useCallback(() => {
```
`React.useCallback` is used via the `React` namespace, which works but is inconsistent with the rest of the codebase that imports `useCallback` directly. More critically, the `useFocusEffect` callback has no cleanup return value when `current?.id` is falsy — it returns `undefined` implicitly instead of a no-op cleanup function.

### `technician/profile.jsx` — Recovery Codes Not Persisted Correctly
`buildRecoveryCodes()` in `profile.jsx` generates codes using `Math.random().toString(36)`, which can produce strings shorter than 12 characters (padded with `"0"`). The spec says codes should be **12 alpha-numeric** characters. The customer OOBE uses `customerSecurityService` for this; the technician profile rolls its own inconsistent implementation.

### `technician/oobe/index.jsx` — Password Not Saved
The `handleSubmit` in the technician OOBE calls `updateUser` with alias, phone, and recovery codes, but **does not include the new password** in the payload:
```caact-mobile/app/technician/oobe/index.jsx#L44-51
      await updateUser({
        ...current,
        alias: alias.trim(),
        phone: phone.trim(),
        recoveryCodes: codes,
        technicianOnboardedAt: new Date().toISOString(),
      });
```
The `password` field is collected but never sent.

### No `tasks` Filter by Unit
The spec says `/tasks` should be filterable by **unit or status**. The implementation only filters by status.

## 4. Customer Portal Issues

### `customer/requests/[id].jsx` and `[id]` Folder Conflict
There is both `app/customer/requests/[id].jsx` and `app/customer/requests/[id]/unit/log/consume-qr.jsx`. Expo Router may have trouble resolving these because `[id].jsx` and the `[id]/` directory share the same dynamic segment name. This should be restructured so `[id].jsx` becomes `[id]/index.jsx`.

### `customer/oobe/reset.jsx` — TOTP Code Not Validated Against Secret
```caact-mobile/app/customer/oobe/reset.jsx#L43-53
  const handleVerify = () => {
    if (!/^\d{6}$/.test(code.trim())) {
      Alert.alert(\"Invalid Code\", \"Enter a 6-digit TOTP code.\");
      return;
    }
    setSubmitting(true);
    Alert.alert(\"TOTP Verified\", ...);
    setSubmitting(false);
  };
```
Any 6-digit number is accepted. The actual TOTP secret is fetched and displayed but never used for validation.

### `customer/settings.jsx` — No Address Picker
The settings screen has a plain `TextField` for the address. The sign-up flow uses the full `BottomSheetSelect` with PSGC municipality/barangay pickers. Customers cannot update their structured address components (municipality, submunicipality, thoroughfare) from settings.

## 5. General Code Quality Issues

### Inline Styles Everywhere
Nearly every screen uses fully inline `StyleSheet`-less style objects. This makes the code verbose, harder to maintain, and prevents style reuse. The spec doesn't mandate this, but it's a significant maintainability concern.

### `__DEV__` Debug Cards Always Visible in Step 2
In `sign-up/step/2.jsx`, the debug card showing the verification code is rendered unconditionally (not gated by `__DEV__`):
```caact-mobile/app/(auth)/sign-up/step/2.jsx#L369-395
        {codeSent ? (
          <Card style={{ ... }}>
            <Text ...>Debug: Verification Code</Text>
            <Text ...>{verificationCode}</Text>
```
This will expose the verification code in production builds.

### `recover/index.jsx` — Redundant Logged-In Check
```caact-mobile/app/(auth)/recover/index.jsx#L24-26
  if (current) {
    return null; // Layout will handle redirect
  }
```
The `(auth)/_layout.jsx` already redirects logged-in users. This check is redundant and returns `null` (a blank screen flash) instead of a `<Redirect>`.

### Missing `useCallback` Dependency Arrays
Several `useFocusEffect` callbacks reference mutable values (like `current`) without proper dependency arrays, or include them but don't account for all dependencies. This is a minor but real source of stale closure bugs.

### `app/_layout.jsx` Still Imports Removed Dependencies
```caact-mobile/app/_layout.jsx#L4-5
import { GestureHandlerRootView } from \"react-native-gesture-handler\";
import \"react-native-get-random-values\";
```
The spec lists both `react-native-gesture-handler` and `react-native-get-random-values` under `dependencies_to_remove` (if not used elsewhere). These are still imported in the root layout.
